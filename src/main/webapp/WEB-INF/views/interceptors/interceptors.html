The specification for CDI Interceptors

An interceptor usually applies an aspect to a CDI Bean such as transactions, logging and so on. Interceptors are ordered either by their definition in the beans.xml or via the @Priority annotation. Interceptors are not ordered by their definition on the intercepted classes. Interceptors are generic and don't know the signature or contract of the intercepted method and should not rely on that at all. Interceptors can share information via the InvocationContext and therefore can ca coupled that way, which should be avoided if possible, or at least the coupling should be as little as possible. Interceptors are enabled on a method or class via an annotation, and therefore a coupled to the intercepted method or class.

See the package at.gepardec.cditraining.interceptors.* for the provided interceptor implementations

1. Interceptor enabled via @Priority

Interceptors enabled with @Priority are enabled globally in all CDI modules. If all interceptors define the same priority then the order they are executed is not deterministic

Your Task:

Ensure that the interceptors declared on the rest resource at.gepardec.cditraining.interceptors.InterceptorsController, are invoked in the following order:

  1. FirstInterceptor
  2. SecondInterceptor
Important!!
Ensure that no interceptor is enabled in the beans.xml and that you restart the server after applying the @Priority annotation.

2. Interceptor enabled via beans.xml

Interceptors enabled with a beans.xml are enabled only in the declaring CDI module. With this approach you have no problems with ordering

Your Task:

Ensure that the interceptors declared on the rest resource at.gepardec.cditraining.interceptors.InterceptorsController, are invoked in the following order:

  1. SecondInterceptor
  2. FirstInterceptor
Important!!
Ensure that no interceptor is enabled via @Priority annotations forehand and that you restart the server after changing the beans.xml.

4. Interceptor with bindings

Interceptors can hold literal values as any other annotation. These values can be used to select an interceptor implementation instead of introducing a new interceptor annotation. By default all annotation attributes are binding and are used for the interceptor resolution. If annotation attributes are not supposed to be considered for the interceptor resolution, then they need to be annotated with @NonBinding.
